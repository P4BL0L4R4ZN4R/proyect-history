Endpoints protegidos con Sanctum y públicos

🔐 Autenticación y Headers Comunes

Las rutas protegidas requieren enviar los siguientes encabezados HTTP en Authorization y Accept para todas las peticiones:

Header Valor Descripción
Authorization Bearer <token> Token de autenticación obtenido tras hacer login
Accept application/json Indica que se espera respuesta en formato JSON

Ejemplo general con cURL

curl -X MÉTODO "https://tuservidor.com/ENDPOINT" \
-H "Authorization: Bearer TU_TOKEN_AQUI" \
-H "Accept: application/json" \
[--data '{ "param": "valor" }']

Reemplaza MÉTODO y ENDPOINT según la operación que realices.

API de Autenticación

Descripción: Endpoints para manejar la autenticación de usuarios, incluyendo login, logout, validación de token y obtención de datos del usuario.

1. Login

POST /api/login_admin

Autentica a un usuario y devuelve un token de acceso.

Ejemplo de petición:
POST https://rutvans.com/api/login_admin
Content-Type: application/json

{
  "email": "admin@example.com",
  "password": "password123"
}
Respuesta exitosa (HTTP 200)
{
  "token": "1|XyZ123...",
  "user": {
    "id": 1,
    "name": "Admin Ejemplo",
    "email": "admin@example.com",
    "roles": [
      {
        "id": 1,
        "name": "admin"
      }
    ],
    "companies": [
      {
        "id": 1,
        "name": "Compañía Principal",
        "role": "admin",
        "status": "active"
      }
    ]
  }
}
Errores comunes:
  • 401 Unauthorized: Credenciales incorrectas
  • 422 Unprocessable Entity: Validación fallida

2. Obtener usuario autenticado

GET /api/user

Obtiene los datos del usuario autenticado.

Ejemplo de petición:
GET https://rutvans.com/api/user
Authorization: Bearer 1|XyZ123...
Respuesta exitosa (HTTP 200)
{
  "id": 1,
  "name": "Admin Ejemplo",
  "email": "admin@example.com",
  "profile_photo_url": "https://rutvans.com/storage/profile-photos/foto.jpg",
  "roles": [
    {
      "id": 1,
      "name": "admin"
    }
  ],
  "companies": [
    {
      "id": 1,
      "name": "Compañía Principal",
      "role": "admin",
      "status": "active"
    }
  ]
}

3. Logout

POST /api/logout

Invalida el token de acceso actual.

Ejemplo de petición:
POST https://rutvans.com/api/logout
Authorization: Bearer 1|XyZ123...
Respuesta exitosa (HTTP 200)
{
  "message": "Sesión cerrada correctamente"
}

4. Validar token

GET /api/validate-token

Verifica si un token de acceso es válido.

Ejemplo de petición:
GET https://rutvans.com/api/validate-token
Authorization: Bearer 1|XyZ123...
Respuesta exitosa (HTTP 200)
{
  "valid": true,
  "message": "Token válido"
}
Respuesta con error (HTTP 401)
{
  "valid": false,
  "message": "Token inválido o expirado"
}

Notas importantes

  • Todos los endpoints excepto login requieren el header Authorization: Bearer [token]
  • Los tokens tienen una duración indefinida (hasta que se haga logout)
  • El endpoint de login devuelve toda la información necesaria del usuario
  • El logout solo invalida el token usado en la petición
  • El token debe incluirse en todas las peticiones autenticadas

  "created_at": "2023-01-01T00:00:00.000000Z",
  "updated_at": "2023-05-01T12:00:00.000000Z"

2. Actualizar perfil

PUT /admin/profile

Actualiza los datos del perfil del usuario autenticado.

Ejemplo de petición (form-data):
PUT https://rutvans.com/api/admin/profile
Content-Type: multipart/form-data

{
  "name": "Admin Actualizado",

  "address": "Nueva Calle 456",
  "phone_number": "9997654321",
  "profile_photo": [archivo de imagen]
}
Parámetros:
CampoTipoRequerido
namestring (max:255)Sí
emailemail (único)Sí
addressstring (max:500)No
phone_numberstring (max:20)No
profile_photoimage (max 2MB)No
Respuesta exitosa (HTTP 200)
{
  "message": "Perfil actualizado correctamente",
  "user": {
    "id": 1,
    "name": "Admin Actualizado",
    "email": "admin.actualizado@example.com",
    "address": "Nueva Calle 456",
    "phone_number": "9997654321",
    "profile_photo_url": "https://rutvans.com/storage/profile-photos/nueva_foto.jpg",
    "profile_photo_path": "profile-photos/nueva_foto.jpg",
    "created_at": "2023-01-01T00:00:00.000000Z",
    "updated_at": "2023-05-02T15:30:00.000000Z"
  }
}

Errores comunes

  • 401 Unauthorized: No autenticado
  • 403 Forbidden: No tiene permisos
  • 422 Unprocessable Entity: Validación fallida

Notas importantes

  • Todas las peticiones requieren autenticación Bearer Token
  • La foto de perfil se almacena en storage/public/profile-photos
  • Al actualizar la foto, se elimina automáticamente la anterior
  • El email debe ser único en el sistema
  • El tamaño máximo de foto es 2MB
  • Los campos address y phone_number son opcionales

5. Eliminar coordinador

DELETE /admin/coordinators/{id}

Elimina el coordinador especificado. También elimina la foto almacenada si existe.

Respuesta exitosa (HTTP 200)
{
  "message": "Coordinador eliminado correctamente"
}

Errores comunes

  • 400 Bad Request: Datos inválidos o faltantes en creación/actualización.
  • 401 Unauthorized: Token no enviado o inválido.
  • 403 Forbidden: Usuario no tiene permiso para la acción.
  • 404 Not Found: Coordinador no encontrado.
  • 409 Conflict: Email duplicado.
  • 500 Internal Server Error: Error en servidor o base de datos.

Notas importantes

  • Los endpoints requieren autenticación mediante token válido (Bearer token).
  • La foto es opcional y puede ser una URL o un path almacenado.
  • El campo site_id debe referenciar un sitio válido existente.
  • El campo employee_code es un identificador único para cada coordinador.

  • Si el usuario no tiene compañías activas, devuelve error 404
